Учебный проект в рамках практического занятия №3 по дисциплине «Технологии разработки приложений на базе фреймворков».  
Реализует веб-сервис на Node.js с гибкой системой конфигурирования, CORS, rate limiting и двумя режимами работы.

## Цели проекта
1. Чтение настроек из трёх источников: файл, переменные окружения, аргументы командной строки. Явный приоритет.
2. Ранняя валидация конфигурации с остановкой запуска при ошибках.
3. CORS-политика: разрешение запросов только с доверенных origin.
4. Ограничение частоты запросов (rate limiting) с раздельными лимитами для чтения списка, получения элемента и создания.
5. Защитные HTTP-заголовки (`X-Content-Type-Options`, `X-Frame-Options`, `Cache-Control`).
6. Два режима: учебный (подробные ошибки) и боевой (минимальная информация).

## Установка и запуск
```bash
# Клонирование репозитория
git clone https://github.com/Dolfi-er/web-service-config-security
cd web-service-config-security

# Установка зависимостей (не требуются, используется только Node.js)
npm install

# Запуск сервера
npm start
```
Сервер стартует на порту, указанном в конфигурации (по умолчанию 3000).

## Обычный запуск
```bash
npm start
```
Используются настройки из config/appsettings.json.
## Запуск с переменными окружения
```bash
APP_MODE=боевой APP_PORT=4000 npm start
```
Доступные переменные окружения:
- APP_MODE — учебный или боевой
- APP_PORT — номер порта (1–65535)
- APP_TRUSTED_ORIGINS — список через запятую (например, http://a.com,https://b.org)
- APP_RATE_LIMIT_LIST — лимит запросов списка в минуту
- APP_RATE_LIMIT_GET — лимит получения одного элемента
- APP_RATE_LIMIT_CREATE — лимит создания элементов
## Запуск с аргументами командной строки
```bash
npm start -- --mode=боевой --port=4000 --trustedOrigins=http://localhost:5173,https://example.com --rateLimitCreate=15
```
Доступные аргументы:
- --mode
- --port
- --trustedOrigins
- --rateLimitList
- --rateLimitGet
- --rateLimitCreate
Аргументы имеют наивысший приоритет.
## Конфигурация и приоритет источников
Итоговая конфигурация формируется в следующем порядке (каждый следующий уровень переопределяет предыдущий):
1. Файл config/appsettings.json (базовые значения)
2. Переменные окружения (префикс APP_)
3. Аргументы командной строки (наивысший приоритет)
## Режимы работы
- учебный (mode: "учебный")
При возникновении ошибки клиенту отправляется подробное сообщение (например, «Имя элемента не может быть пустым»). Полезно при отладке.
- боевой (mode: "боевой")
Клиент получает обезличенное сообщение «Ошибка обработки запроса». Подробности пишутся только в логи сервера (в данной реализации — в консоль).
## Критичные настройки и безопасность
При запуске выполняется строгая валидация конфигурации. При обнаружении ошибок сервер не стартует, а в консоль выводится список проблем.

Проверяемые параметры:
- mode — только "учебный" или "боевой".
- port — целое число от 1 до 65535.
- trustedOrigins — непустой массив URL с протоколами http: или https: и без пути (только origin). Это предотвращает случайные «дыры» из-за опечаток.
- rateLimits:
-- list, get, create — целые положительные числа.
-- Лимит создания (create) не может превышать лимит списка (list).

Защитные меры во время работы:
- CORS – запросы принимаются только от origin, перечисленных в trustedOrigins.
- Rate limiting – скользящее окно 60 секунд, раздельные счётчики для маршрутов:
-- GET /api/items (list)
-- GET /api/items/by-id/:id (get)
-- POST /api/items (create)
- Заголовки безопасности:
-- X-Content-Type-Options: nosniff
-- X-Frame-Options: DENY
-- Cache-Control: no-store, max-age=0
## API сервиса
Базовый URL: http://localhost:<port>
### Получить текущий режим
```bash
GET /api/mode
```
### Получить список элементов
```bash
GET /api/items
```
### Получить элемент по ID
```bash
GET /api/items/by-id/:id
```
### Создать элемент
```bash
POST /api/items
Content-Type: application/json

{
  "name": "Название",
  "price": 100
}
```

Тесты проверяют:

- Валидацию некорректных настроек

## Примечание
- Для хранения данных используется in-memory репозиторий. При перезапуске сервера все созданные элементы теряются.
- Идентификаторы генерируются с помощью crypto.randomUUID().
- При превышении лимита запросов сервер возвращает статус 429 и заголовок Retry-After.
- Проект использует только встроенные модули Node.js, дополнительные зависимости не требуются
